# W0/W1 验收矩阵与证据协议

## 1. Evidence Lanes

`evidence_lane` 是当前机器字段，表示证据单元类型；它不是 workload benchmark 分类。

| Evidence lane | Machine value | 数据 | 可提交内容 | 用途 | 不可替代 |
|---|---|---|---|---|---|
| smoke | `smoke` | synthetic/public tiny set | pass/fail, counts, redacted summary | 快速防回归 | 不代表性能基线 |
| W0 | `w0_docs` | public/synthetic + docs checks | commands, diff scope, guard output | PR 可审查证据 | 不代表私有真实语料 |
| W1 | `w1_private` | local private resumes + local private query set | public-safe identifiers, counts, percentiles, resource aggregates | 真实本机性能证据 | 不提交 raw data 或 raw local hashes |
| soak/fault | `soak_fault` | local controlled fault run | redacted aggregate, failure class, recovery status | 长跑和恢复证据 | 不代表 query quality |
| GUI/manual | `gui_manual` | local GUI + daemon IPC | screen-state checklist, redacted diagnostics flags | 人机闭环体验 | 不代表 index scalability |

## 1.0 Synthetic Smoke Baseline

P1 的无私有数据起点是 `smoke` lane 的 synthetic baseline。它只证明
benchmark harness、redacted report、artifact manifest、contract pins 和 privacy
guards 能闭环运行；它不是 W1、D10K、D100K、D1M、profile optimization issue、
scale gate 或 `goal_complete` 证据。

Synthetic smoke report 必须使用：

```text
schema_version = resume-ir.experiment-report.v2
report_kind = redacted_evidence
claim = no_claim
evidence_lane = smoke
synthetic_smoke.smoke_schema_version = resume-ir.synthetic-smoke-baseline.v1
manifest schema = resume-ir.synthetic-smoke-artifact-manifest.v1
```

`contract_pins.git_head_sha` must be an actual local git commit for smoke
report/manifest evidence; the `working-tree` placeholder is invalid for this
lane.

Allowed command shape is exact:

```text
resume-benchmark synthetic-query --index-dir <redacted-temp-index> --documents 24 --queries 6 --top-k 5 --json
```

`--index-dir <redacted-temp-index>` is the only optional redacted path field in
the public command string. The `--documents`、`--queries` 和 `--top-k` values
must match the report counters. Any synthetic report containing
any non-smoke top-level evidence field, D10K/D100K/D1M scale field,
W1/private lane claim, profile optimization claim, scale-gate claim or
`goal_complete` claim must fail closed.
Report and manifest are inseparable: setting only
`RESUME_IR_BENCHMARK_SMOKE_REPORT_OUT` or only
`RESUME_IR_BENCHMARK_SMOKE_MANIFEST_OUT` is invalid, and standalone smoke
reports must fail `check-experiment-report.py` unless their matching manifest is
validated in the same invocation.
Because this lane is the public harness baseline, `synthetic_smoke.latency_ms`
must also carry `batch_protocol_stage` timings parsed from the actual
`benchmark-query-protocol --batch-jsonl` smoke output. This proves the batch
protocol and stage timing surface ran without turning synthetic smoke into W1,
D10K, profile, scale, or `goal_complete` evidence.
The same smoke harness must also exercise the `resume-benchmark private-query`
runner path with a synthetic/private-shaped query-set so the runner-to-resident
batch command contract is covered before any private D10K calibration; this
runner smoke remains public synthetic evidence and cannot publish real query
digests or satisfy W1/D10K.

## 1.1 Benchmark Lanes

`benchmark_lane` 是 workload/measurement 分类，和 `evidence_lane` 正交。Benchmark evidence is split into five required lanes. Lanes cannot claim each other's results.

`benchmark_lane` is schema-backed in `perf/experiment-report.schema.json` for
redacted experiment reports. `perf/current-loop-state.json` still keeps only
`evidence_lane` as the current loop snapshot field; workload-lane completion is
proved by lane-specific reports and issue/PR evidence, not by a parallel
current-state field.

| Benchmark lane | Purpose | Allowed claim boundary |
|---|---|---|
| `first_searchable` | prove how quickly directly parseable documents become searchable | first searchable hot path only; OCR completion is not a pass condition |
| `full_import_ocr_backlog` | prove full import, OCR queue behavior, external OCR isolation, recovery, and background scheduling | OCR backlog and scheduler behavior only; cannot claim query P95 success |
| `query_hot_path` | prove hot-index query performance under a resident daemon | resident hot query latency only; no OCR, parsing, heavy model, or spawn-per-query claim |
| `agent_query_replay` | prove behavior under a static frozen real SeekTalent Agent query workload | real agent workload latency and parser compatibility only; cannot claim human search quality |
| `repeat_amplification_control` | detect dedupe, cache, index, and duplicate-result pathologies | pathology detection only; cannot claim real distribution quality |

### 1.1.1 Mixed Import Product-Correctness Evidence

Mixed import correctness is a product-contract precondition before GUI and
query hot-path optimization. It is recorded under the existing evidence lanes
(`smoke` for public synthetic, `w1_private` for private calibration/holdout)
and does not by itself satisfy D10K/D100K/D1M scale gates.

Required layers:

| Layer | Data | Public output | Purpose |
|---|---|---|---|
| `public_synthetic` | committed exact synthetic mixed fixture | bounded aggregate report | CI regression and schema proof |
| `private_calibration` | local authorized mixed nested corpus | redacted aggregate only | classifier tuning and before/after comparison |
| `blind_holdout` | local authorized holdout not used for tuning | redacted aggregate only | anti-overfit acceptance |

Public synthetic opaque fixture IDs and exact labels are visible only in the
committed synthetic test source. They are not classifier features and must not
be copied into report output; public reports remain bounded aggregates. Private
calibration and blind-holdout per-file labels remain local-only.

Required metrics:

1. sample counts by label, extension bucket, size bucket, and directory-depth
   bucket;
2. `resume_candidate`、`non_resume`、`needs_review`、`ocr_backlog`、`failed`
   counts;
3. indexed resume precision, contamination count, resume completeness;
4. clean-vs-mixed wall time and stage timings;
5. H-tier/resource budget and privacy flags.

No mixed import report may expose raw source paths, filenames, raw text, raw
labels, candidate results, raw query text, direct raw file hashes, screenshots,
tokens, diagnostics packages, SQLite/index/sidecar data, or private manifests.

## Platform Lane Evidence

`macos_m4_discovery` 可用于 hotspot 排序、profile hypothesis 和快速回归，但不能单独关闭 Windows weak-host、cross_os 或 goal_complete gate。

`windows_weak_host_validation` 在以下节点是必需证据：

1. L1 architecture acceptance，若 claim 涉及 lifecycle、filesystem watching、recovery 或 first-searchable。
2. L2 parallelism acceptance。
3. GUI default-stack packaging 或 blocker-driven fallback acceptance。
4. D100K weak-host acceptance。
5. `cross_os_passed`。
6. `goal_complete`。

`cross_os_ci_smoke` 不能替代 `windows_weak_host_validation`。

## 2. W0 Docs-hardening Gate

当前 docs-hardening PR 只需要 W0：

```bash
python3 scripts/ci/check-performance-contracts.py
python3 -m py_compile scripts/ci/check-performance-contracts.py
git diff --check -- GOAL.md MANIFEST.md ACTIVE_GOAL.toml docs/superpowers 03_next_goal_高性能本地检索GUI闭环 perf scripts/ci/check-performance-contracts.py .github/workflows/pr.yml
git diff --check --cached -- GOAL.md MANIFEST.md ACTIVE_GOAL.toml docs/superpowers 03_next_goal_高性能本地检索GUI闭环 perf scripts/ci/check-performance-contracts.py .github/workflows/pr.yml
while IFS= read -r file; do
  tmp="$(mktemp)"
  git diff --check --no-index /dev/null "$file" >"$tmp" 2>&1
  rc=$?
  if [ -s "$tmp" ]; then
    cat "$tmp"
    rm "$tmp"
    exit 1
  fi
  rm "$tmp"
  if [ "$rc" -ne 0 ] && [ "$rc" -ne 1 ]; then
    exit "$rc"
  fi
done < <(git ls-files --others --exclude-standard '*.md')
./scripts/ci/guard-public-repo.sh
{
  git diff --name-only origin/main...HEAD
  git diff --name-only --cached
  git diff --name-only
  git ls-files --others --exclude-standard
} | sort -u
```

Expected:

1. diff check passes.
2. public repo guard passes.
3. JSON/TOML machine contracts parse and satisfy `scripts/ci/check-performance-contracts.py`.
4. valid fixtures under `perf/fixtures/valid/` pass, invalid fixtures under `perf/fixtures/invalid/` fail.
5. changed paths stay inside the approved docs + public-contract-gate scope.

## 3. Scale Benchmark Gate

后续性能实现完成后，redacted evidence 必须同时满足数据规模、语义、性能、热路径和隐私红线。阈值源文件是 `perf/acceptance-matrix.toml`。

规模分三档：

| Gate | 用途 | 最小文档 | 最小可搜索 | 最小 query | 最小 request samples | 能否作为 scale gate 的完成候选 |
|---|---|---:|---:|---:|---:|---|
| D10K_private_calibration | 首个真实私有校准和正确性基线 | 10000 | 8000 | 500 | 5000 | 否 |
| D100K_weak_host | 弱机/节能/调度风险证明 | 100000 | 90000 | 500 | 10000 | 否 |
| D1M_scale | 完整性能目标规模 | 1000000 | 900000 | 500 | 25000 | only with all other evidence lanes and benchmark lanes |

D10K/D100K 失败必须生成 blocked 或 failed redacted aggregate；D10K/D100K 通过也不得绕过 D1M、soak/fault、GUI/manual 和五个 required benchmark lanes 直接完成目标。

### 3.1 Required Evidence Fields

1. query_set_sha256。
2. query_count。
3. bucket_counts。
4. document_count。
5. searchable_document_count。
6. vector_indexed_document_count。
7. machine_profile。
8. resident_daemon_batch_id。
9. spawn_per_query=false。
10. p50/p95/p99 per bucket。
11. per-record elapsed_ms source for query latency percentiles。
12. histogram_bins per stage and bucket。
13. stage latency per bucket。
14. rss_peak_mb、cpu、disk read/write aggregate。
15. profiler_capture_refs：public-safe redacted symbol-summary/report hash、`benchmark_report_hash`、`benchmark_artifact_id`、approved opaque manifest ref 或 HMAC-SHA256 opaque manifest ref；raw local profiler capture/file hashes 不得进入 git/GitHub。
16. trace_summary_redacted=true。
17. hot_path_ocr=false。
18. hot_path_parsing=false。
19. hot_path_heavy_model_inference=false。
20. percentile_confidence。
21. holdout_sha256：public-safe identifier only, not a raw holdout query-list hash。
22. thresholds.passed。
22. failed_redlines。
23. contract_pins：active goal、acceptance matrix、loop schema、experiment schema 和 git head。

### 3.2 Redlines

| 类别 | 红线 |
|---|---|
| dataset | 按 D10K/D100K/D1M gate 满足最小 `document_count` 和 `searchable_document_count` |
| query set | `query_count >= 500`, request sample count 满足 gate，public-safe tune/holdout identifiers 均存在；raw local query-set/holdout hashes local-only |
| bucket set | `single_term`, `and_2`, `and_3_5`, `and_6_16`, `field_filter`, `hybrid`, `semantic` 均存在 |
| runner | `resident_daemon_required=true`, `spawn_per_query=false` |
| hot path | OCR、parsing、heavy model inference 均为 false |
| single_term | D1M P95 <= 100ms, P99 <= 220ms |
| and_2 | D1M P95 <= 120ms, P99 <= 250ms |
| and_3_5 | D1M P95 <= 140ms, P99 <= 280ms |
| and_6_16 | D1M P95 <= 180ms, P99 <= 360ms |
| field_filter | D1M P95 <= 160ms, P99 <= 320ms |
| hybrid | D1M P95 <= 200ms, P99 <= 400ms |
| semantic | D1M P95 <= 400ms, P99 <= 800ms |
| stage | parse <= 5ms, prefilter <= 20ms, bm25 <= 80ms, ann <= 180ms, fusion <= 15ms, hydrate <= 60ms, snippet <= 60ms |
| zero-change incremental | `reparse_count=0`, `full_rebuilds=0`, `changed_manifest_rows=0` |
| privacy | raw resume、raw query、candidate、path、token、diagnostics package 均为 false |

任一红线未达标时，报告可以提交为 blocked/failed redacted aggregate，但不得声明 scale gate accepted。阈值变更必须经过 Superpowers planning 和 local gstack-lite plan/readiness review，不得由实现代码或 benchmark runner 临时覆盖。

## 4. Soak and Fault Gate

Long-run evidence must cover:

1. resident daemon search batch。
2. daemon restart during import。
3. cancel active request。
4. overload response。
5. disk-space-low。
6. file-lock。
7. snapshot corrupt。
8. journal gap reconciliation。
9. degraded low-resource mode。

## 5. GUI/manual Gate

GUI acceptance must cover:

1. import submit。
2. status levels。
3. keyword search。
4. field filter search。
5. hybrid search partial/degraded display。
6. detail view 通过 authenticated local detail/hydrate 自动显示本地正文和 display path，不直读内部存储。
7. diagnostics export redaction flags。
8. pause/resume/cancel。
9. overload display。
10. benchmark lane display。

GUI visual/packaging/performance acceptance threshold：

默认 GUI stack 是 `Tauri + React + Vite + Tailwind + TypeScript`。bakeoff 只在已链接 GitHub blocker issue 证明默认 stack 无法满足 visual、packaging 或 performance contract 时作为 fallback 适用，不是默认验收门槛。

1. representative model 至少模拟 100000 logical results。
2. 1440x900 参考搜索态稳定渲染恰好 4 张与 `UI-reference/search.png` 密度一致的大结果卡。
3. 导航可使用滚动或分页；滚动 buffer 最多 8 张，scroll/update 至少 10Hz，hover/loading/selection 不改变卡片高度。
4. daemon mock 或本地 daemon 的 interactive search 至少 10 qps，不阻塞 UI。
5. desktop/tablet/narrow viewport 均必须保留 search、result count、partial/overload reason 和 export status。

## 6. Completion Rule

完整 performance + GUI goal 只有在 W0、D10K、D100K、D1M、soak/fault、GUI/manual 均有对应 redacted evidence，且 `first_searchable`、`full_import_ocr_backlog`、`query_hot_path`、`agent_query_replay`、`repeat_amplification_control` 五个 required benchmark lanes 均有对应 redacted evidence，且 `scripts/ci/check-performance-contracts.py` 的负 fixtures 继续失败、正 fixtures 继续通过后，才能标为 complete。

## 7. S807 correctness/recovery acceptance cell

The v27 correctness cell is prerequisite evidence, not a performance result.
It requires: deterministic RST coverage for status/search/batch/detail/hydrate/
progress; daemon survival and exact bounded counters after 100 mixed connection
faults; native supervisor recovery/circuit/heartbeat/normal-exit behavior;
persistent redacted lifecycle receipts readable while the daemon is dead;
content A-to-B producing distinct immutable versions; all-old/all-new behavior
at every publication failure point; exact selection semantics across unrelated
epoch changes, target replacement, delete, and paged hydrate races; agreement
of fulltext/field/vector/hybrid doc-version pairs; v26 copy-on-write migration
remaining repairing until validated rebuild; and UI snapshots for recovering,
circuit-open, blocked, stale-selection, and interrupted-detail states.

Synthetic soak must run for at least 120 minutes. macOS installed-App evidence
must prove forced daemon recovery and zero process residue on normal App exit.
Windows build-host compilation is useful evidence but cannot replace a native
H0 Job Object/installer lifecycle run after reset or reprovision. Public output
is bounded redacted aggregate only and contains no PID, path, token, query,
resume body, raw stderr, runtime receipt, or diagnostics package.
