# 数据流与状态机

## 1. 首次导入快模式

```mermaid
flowchart LR
    A["Start watcher first"] --> B["Scan roots"]
    B --> C["Source gate + stable identity"]
    C --> D["Quick fingerprint"]
    D --> E["Parse text layer"]
    E -- failed --> X["failed local record"]
    E --> F{"text available?"}
    F -- no --> G["ocr_backlog local record"]
    G --> H["OCR queue"]
    H --> I["OCR text extraction"]
    I --> J["Normalize + sectionize"]
    F -- yes --> J
    J --> K["Resume classifier"]
    K --> L{"classification status"}
    L -- resume_candidate --> M["Field extract best-effort"]
    M --> N["SQLite batch prepare"]
    N --> O["Fulltext + fields commit"]
    O --> P["Visible Level1/2"]
    P --> Q["Semantic queue"]
    L -- non_resume --> R["Local audit record"]
    L -- needs_review --> S["Local review backlog"]
    L -- failed --> X
    R --> T["No resume index write"]
    S --> T
    X --> T
```

快模式目标是先让可解析文本进入可搜索状态。OCR 和 semantic 是后台增强，不阻塞 Level1/2 可见性。

## 1.1 Mixed Import Classifier Placement

`fs-crawler` 只做 cheap source discovery：root 遍历、ignore 规则、支持扩展名、
权限、大小、mtime、fingerprint 和稳定身份。它不能把文件名、目录名或扩展名
当成最终“是不是简历”的产品语义。

真正的 resume classifier 必须位于 parse/text extraction 与 searchable index
写入之间：

```text
scan / extension gate
-> parse / text extraction
-> if text missing: ocr_backlog local record, OCR queue, OCR 后回到 text extraction
-> normalize / sectionize
-> resume classification
-> resume_candidate: DB searchable + full-text/field index
-> non_resume: local audit record only
-> needs_review: local review backlog only
-> ocr_backlog: OCR queue, OCR 后再分类
-> failed: local failure record only
```

分类器第一阶段不使用 AI/LLM。它必须是确定性、多信号、可解释的
precision-first gate，且禁止单一信号直接放行：文件名、目录名、扩展名、
联系方式、标题或单个关键词都不能单独决定 `resume_candidate`。

## 4. Level 可见性

| Level | 含义 | 用户可见能力 |
|---|---|---|
| Level1 | 文件和基础元数据可见 | GUI 看到文件已发现、状态可查 |
| Level2 | 文本层全文可搜 | keyword、filename、基础 snippet 可用 |
| Level3 | 字段/semantic/OCR 增强完成或部分完成 | filter、hybrid、semantic、OCR 内容逐步增强 |

GUI 必须明确展示 Level，而不是把“导入完成”误写成“所有 OCR/semantic 完成”。

## 5. 后台预算状态

| 状态 | 触发 | 动作 |
|---|---|---|
| `interactive` | 用户正在查询或 GUI 活跃 | 降低 OCR/vector/merge |
| `balanced` | 默认接电/SSD/资源正常 | 常规后台吞吐 |
| `energy_saver` | 电池、热、低内存、HDD | 暂停 OCR/vector，保留查询 |
| `repair` | 检测到派生状态缺失或损坏 | 局部 rebuild，不全库重建 |

Profile names used by implementation and evidence reports:

Canonical profile names are `H0_Eco`, `H1_Balanced`, and `H2_Aggressive`.
Non-canonical profile labels are retired and must not appear in implementation,
diagnostics, or public-safe evidence.

RAM determines the static profile and private/anonymous memory budget. CPU or
available parallelism only caps the effective worker count; it does not
downgrade a high-memory host into a lower memory-budget tier. Disk stall,
system memory pressure, battery, and thermal signals remain dynamic governor
inputs below, not static profile selectors.

| Profile | RAM threshold | Import parser workers | Private/anonymous budget | 后台策略 |
|---|---|---:|---:|---|
| `H0_Eco` | unknown or <= 12 GiB | 1, capped by available parallelism | 512 MiB | OCR/vector/import 严格限流，batch 小，必要时只做 Level1/Level2 |
| `H1_Balanced` | > 12 GiB and <= 20 GiB | 2, capped by available parallelism | 1024 MiB | 后台正常吞吐，遇到 overload 先暂停 benchmark/background |
| `H2_Aggressive` | > 20 GiB | 3, capped by available parallelism | 1536 MiB | 允许更高 OCR/vector/import 并发，但需要同样的 redacted aggregate evidence |

## 6. Adaptive Governor Contract

低配和节能模式必须由可观测信号驱动，不允许只写成“后台降级”口号。

| Signal | Threshold | Action | Recovery |
|---|---|---|---|
| interactive latency | P95 > 200ms 连续 3 个 10s window | OCR/vector/import 并发减半，benchmark 降速 | P95 <= 150ms 连续 6 个 window |
| interactive overload | P95 > 400ms 或 queue delay > 200ms | 暂停 benchmark/background，必要时返回 partial/`OVERLOADED` | 6 个 window 正常且 dwell >= 60s |
| RSS / memory pressure | private/anonymous RSS 超过当前 `H0_Eco`/`H1_Balanced`/`H2_Aggressive` 预算，或系统 memory pressure elevated | 清低优先级 cache、暂停 heavy background、保留 resident reader | private/anonymous RSS 回到当前 profile 预算内 6 个 window |
| page fault / disk stall | major faults > 100/s 或 fsync P95 > 100ms | 暂停 compaction/snapshot/import 大批次，禁止整包 publish | IO 指标恢复 6 个 window |
| battery / thermal | battery < 20% 或 thermal serious | 进入 `energy_saver`，只保留必要 indexing 和前台查询 | 接电或 thermal normal，dwell >= 60s |
| HDD / slow media | 随机读写或 fsync latency 超阈值 | 减小 batch、延后 vector/OCR、优先 manifest diff | media 指标恢复 6 个 window |

每次状态切换必须进入 daemon status、redacted diagnostics 和 GUI 可见原因，并进入 W1/soak evidence 的 resource aggregate。阈值源文件是 `perf/acceptance-matrix.toml`；实现期如需调整，必须改矩阵并重新 review，不能在代码中静默漂移。

## 11. S807 v27 publication and recovery state machines

Import publication is now:

```text
discover -> stage SourceRevision/ResumeVersion/derived facts
-> build fulltext+vector snapshots -> validate snapshots
-> CAS ActiveSearchProjection + terminal ingest + generations + visible epoch
-> published
```

Until the CAS succeeds, the previous projection remains readable. Retryable
parse/index failures do not demote the active version. Confirmed delete,
permanent exclusion, and privacy revocation remove admission through the same
atomic publication boundary. Snapshot GC waits for all pinned query leases.

Desktop lifecycle is independent of service health:

```text
starting -> ready -> recovering -> ready
                  -> circuit_open -> half_open
                  -> blocked
```

Only the native supervisor owns the child. A restart changes generation and
interrupts in-flight requests; it never replays search, import, detail, or
hydrate. Service health (`ready/degraded/repairing`) and result freshness
(`current/stale/interrupted`) remain orthogonal to lifecycle.
