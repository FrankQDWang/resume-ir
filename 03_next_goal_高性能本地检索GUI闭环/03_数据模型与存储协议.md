# 数据模型与存储协议

## 1. Manifest 主模型

Manifest 是下一阶段的根数据模型。

| 字段 | 作用 | 说明 |
|---|---|---|
| `file_pk` | 内部主键 | 单机递增 |
| `stable_file_id` | 稳定文件身份 | Windows 使用 volume + file id；macOS/POSIX 使用 dev + ino + birthtime 辅助 |
| `root_id` | 来源 root | root 可 offline、disabled、rescanning |
| `display_path` | 用户展示路径 | 仅本地显示，不进入公开证据 |
| `canonical_path_hash` | 路径哈希 | 诊断和匹配用 hash，不提交 path |
| `size_bytes` | 文件大小 | quick diff |
| `mtime_ns` | 修改时间 | quick diff |
| `birthtime_ns` | 创建时间 | inode/fileid 回收防护 |
| `quick_fingerprint` | 快速内容指纹 | 判断是否值得解析 |
| `strong_hash` | 强内容哈希 | 按需计算，支持跨路径识别 |
| `file_kind` | 文件类型 | pdf/docx/doc/txt/image/unsupported |
| `parse_state` | 解析状态 | state machine 输入 |
| `text_epoch` | 文本可见 epoch | 文本层搜索可用性 |
| `fulltext_epoch` | 全文索引 epoch | BM25 可用性 |
| `field_epoch` | 字段索引 epoch | filter 可用性 |
| `vector_epoch` | 向量 epoch | semantic 可用性 |
| `snippet_epoch` | snippet sidecar epoch | 结果片段可用性 |
| `last_seen_scan_epoch` | 最近扫描轮次 | 删除和 offline 判断 |
| `tombstone` | 逻辑删除 | 查询默认隐藏 |
| `retry_after_ts` | 重试时间 | 退避 |
| `error_code` | 最近错误码 | diagnostics 和 GUI 展示 |
| `parser_version` | 解析器版本 | 版本变更触发局部重解析 |
| `schema_version` | 存储/索引 schema | breaking change 门槛 |
| `resume_classification` | mixed import 分类状态 | `resume_candidate` / `non_resume` / `needs_review` / `ocr_backlog` / `failed` |
| `classification_reasons` | 分类解释码 | 本地 audit/review 用；公开证据只能输出 reason bucket count |
| `classification_epoch` | 分类版本 | classifier 版本或阈值变化触发局部重分类 |
| `review_state` | 人工复核状态 | 仅本地 backlog，不进入公开证据 |

## 1.1 Mixed Import Classification Record

Mixed import 的分类状态是产品语义合同，不是 parser 成功与否的别名：

| 状态 | 搜索索引 | 本地记录 | 含义 |
|---|---|---|---|
| `resume_candidate` | yes | yes | 多信号高置信简历，可进入 resume search index |
| `non_resume` | no | yes | 明显非简历，保留 audit 记录和聚合原因 |
| `needs_review` | no | yes | 证据不足或冲突，进入本地 review backlog |
| `ocr_backlog` | no | yes | 扫描件/无文本/需 OCR；OCR 前不得判成 `non_resume` |
| `failed` | no | yes | 权限、格式、解析或 IO 失败 |

逐文件 audit/review 记录可在本地存储 `document_id`/fingerprint、原始路径、
扩展名、大小、分类状态、reason codes、分类时间和 parser/OCR 状态。公开报告、
GitHub issue、PR、benchmark summary 和 committed fixtures 不得包含真实路径、
文件名、正文、直接 raw file hash 或私有 labels。

## 2. Mutation Batch

所有导入和增量更新都必须经过 batch 协议：

```text
prepare(batch_id, root_id, manifest_delta, target_epoch)
apply_fulltext(batch_id)
apply_fields(batch_id)
apply_snippet_sidecar(batch_id)
apply_vector(batch_id)
commit(batch_id, visible_epoch, commit_ids)
gc(compactable_before_epoch)
```

## 3. 恢复规则

| 状态 | 恢复动作 |
|---|---|
| prepared 但未 apply | 重放 batch |
| fulltext 已 apply 但 SQLite 未 commit | 用 `batch_id` 幂等对账 |
| SQLite 已 commit 但派生层缺失 | 从 batch log 补写派生层 |
| vector 失败 | semantic partial，keyword/field 仍可用 |
| snippet sidecar 缺失 | 返回无 snippet 或短 fallback，不阻塞搜索 |
| tombstone 长期存在 | 等旧 reader drain 后物理清理 |

## 4. Visible Epoch

Visible epoch 是 GUI、CLI、daemon 和 Codex 验证共同理解系统状态的唯一版本点。

每个 response 至少能说明：

1. `visible_epoch`
2. `fulltext_epoch`
3. `field_epoch`
4. `vector_epoch`
5. `snippet_epoch`
6. `partial_reason`

## 5. Sidecar

正文和长 snippet 不进入 Tantivy 热路径 STORED 大字段。

| Sidecar | 内容 | 访问方式 |
|---|---|---|
| snippet sidecar | top result 片段、短摘要、highlight window | doc id -> pointer -> chunk |
| body sidecar | 规范化正文和 section chunk | detail 或 snippet fallback 才读 |
| vector sidecar | embedding payload、ANN snapshot、tombstone | semantic layer 读取 |

写入采用同卷 temp + 校验 + atomic replace/rename。

## 6. Stable Identity and Rename Contract

稳定身份必须拆成三层，避免路径变化、inode 回收或重复文件让增量索引漂移：

| 层 | 字段 | 合同 |
|---|---|---|
| file identity | `stable_file_id` | Windows 使用 volume + file id；macOS/POSIX 使用 dev + ino + birthtime 辅助，不能只依赖 path |
| content identity | `quick_fingerprint` + `strong_hash` + `content_fingerprint` | quick 用于热路径判断，strong/content 用于跨路径去重、rename 识别和 reparse 决策 |
| path alias | `path_alias` / `canonical_path_hash` | path 是展示和定位 alias，不是业务事实主键；公开证据只能使用 hash |

Rename handling:

1. 同一 `stable_file_id`、同一 content fingerprint、不同 path alias，视为 rename，不触发全文重建。
2. 同一 path alias、不同 `stable_file_id` 且 content fingerprint 不同，视为 replacement，进入 mutation batch。
3. inode/file id 回收必须通过 birthtime、size、mtime 和 content fingerprint 防护。
4. root offline 不批量 tombstone；必须等 root 重新 online 或完成 bounded reconciliation。

## 12. S807 v27 immutable publication hard cut

Schema v27 replaces mutable resume rows with three explicit facts:

1. `Document` is stable source identity and reachability only.
2. `SourceRevision` is content-addressed source evidence; `ResumeVersion` is an
   insert-once value derived from document identity, source/content revision,
   normalized-text hash, parser version, and schema version. The same version id
   with different bytes or derived facts is an invariant failure.
3. `ActiveSearchProjection(document_id -> resume_version_id)` is the sole search
   admission fact. Classification, mentions, candidate linkage, full-text rows,
   and vector rows are version-bound; mutable visibility is retired.

Staged revisions and derived data are inaccessible to query/detail. Publication
first validates complete full-text and vector snapshots, then one SQLite CAS
transaction swaps the projection, ingest terminal state, index generation,
visible epoch, and publication journal. A query lease pins immutable readers,
metadata read transaction, generation, and epoch so a request is all-old or
all-new. v26 migration is copy-on-write: preserve only authorized root/source/
document identity, invalidate all legacy derived facts and index artifacts, and
remain `repairing` until a validated v27 publication exists. No dual reader,
legacy alias, synthetic version backfill, or latest-row fallback is allowed.
