# 目标边界与成功标准

## 1. 目标

下一阶段目标是一次性完成高性能本地检索目标包：

1. 建立高性能目标文档、系统契约和验收门槛。
2. 用本地私有简历库完成导入、热索引、查询和增量 benchmark。
3. 用 SeekTalent artifacts 中的 query 与 query 组合形态构建私有 query benchmark 种子。
4. 完成 manifest、stable file identity、dirty subtree、mutation batch、visible epoch 的数据面设计和实现。
5. 优化热查询路径：字段前置过滤、fast-field/bitmap、bulk hydrate、snippet/body 侧车、daemon cache。
6. 把 OCR、semantic、vector 构建改为可预算、可暂停、可降级的后台增强层。
7. 建立 GUI/manual usage 与 Codex 结对闭环验证：本地导入、状态、搜索、详情、诊断、性能证据。

## 1.1 当前优先级：mixed import product correctness

#132-#137 已经证明 import L1/L4 优化进入边际收益变薄阶段。当前 `03`
目标不另起新目录，但执行顺序先从 mixed-directory import 的产品语义补齐开始：

1. 完成 #138 只读 product-capability audit，确认混合目录导入、搜索召回和
   detail/hydrate 的真实合同缺口。
2. #140 先冻结 public synthetic、private calibration 和 blind holdout 的合同；
   后续 classifier/private-calibration linked issue 才可在本地构造真实 witness：
   以已知正例的干净私有简历 corpus 为 resume 标签来源，再从用户授权的
   `$RESUME_IR_MIXED_SOURCE_ROOT` 加入非简历和干扰文档，保留多层目录结构。
3. 先实现 precision-first classifier：被索引的文档必须尽可能都是简历；
   不为了提高自动召回而让非简历污染 resume search index。
4. 在 precision 和 contamination 红线保持稳定后，再提高
   resume completeness/recall，减少真实简历落入 `needs_review` 的比例。
5. mixed import 语义稳定后，重新 profile mixed import，再决定是否进入 L2
   worker、queue、backpressure、DB/index 写入协调和资源预算优化。
6. GUI 和 query hot path 可以审计合同，但不得在 mixed import 语义和证据合同
   稳定前大规模实现或优化。

## 2. 非目标

本目标不包含：

1. GitHub Release 发布。
2. Apple/Windows 真实签名、公证、证书管理。
3. 上传真实简历、raw query、trace、SQLite、索引、诊断包或模型缓存。
4. 把 SeekTalent 候选人、结果卡、公司、履历、页面 trace 迁入本仓库。
5. 为老数据格式做向后兼容 shim。产品未发布，允许 breaking change。
6. 把 field extraction 或 dedupe 当成完成本目标的默认 gate；它们是检索增强能力，不是是否可搜索的根门槛。
7. 在 mixed import 语义稳定前实现 GUI、优化 query hot path，或继续新的 L4
   import 微优化。

## 3. 成功标准

### 3.1 文档完成

目标文档必须覆盖：

1. 架构和模块边界。
2. 数据模型和存储协议。
3. 导入、增量、查询、GUI 的数据流。
4. daemon IPC 和 diagnostics contract。
5. query benchmark 的私有来源、脱敏、分桶和验收。
6. 失败模式、恢复策略、安全隐私边界。
7. 实施切片、测试命令、退出标准。
8. 公开机器合同校验：`python3 scripts/ci/check-performance-contracts.py` 必须能在 CI 中证明矩阵、schema、fixtures 和隐私布尔字段自洽。

### 3.2 性能完成

在用户授权的本地私有语料上，至少生成 redacted aggregate evidence：

1. 首次导入快模式：TTF100、TTF1000、searchable count、RSS、错误分类必须输出，并标明 Level1/Level2 是否不等待 OCR/semantic。
2. 零变更增量：`reparse_count=0`、`full_rebuilds=0`、`changed_manifest_rows=0`，否则不得声明 true incremental 达标。
3. 小比例 churn：1%、5%、10% 变更只允许重建 changed files 加 dirty subtree，禁止 full rebuild 伪装成增量。
4. 查询 benchmark 分三档：D10K 私有校准、D100K 弱机规模、D1M 完成规模。D10K/D100K 可以作为校准和风险证据，但不得声明完整 `goal_complete`。
5. D10K calibration 至少 `document_count >= 10000`、`searchable_document_count >= 8000`、`query_count >= 500`、`request_sample_count >= 5000`，resident daemon，`spawn_per_query=false`。
6. D100K weak-host 至少 `document_count >= 100000`、`searchable_document_count >= 90000`、`request_sample_count >= 10000`，用于证明低配/节能策略和调度不只在 10k 上成立。
7. D1M scale 至少 `document_count >= 1000000`、`searchable_document_count >= 900000`、`request_sample_count >= 25000`，并达到 `perf/acceptance-matrix.toml` 中 per bucket P95/P99/stage 红线，才可作为完整性能目标证据。
8. 低配/节能模式：adaptive governor 必须按阈值降级后台任务，interactive keyword/field search 不得超过 IPC deadline 后继续阻塞。
9. 热路径证明：`hot_path_ocr=false`、`hot_path_parsing=false`、`hot_path_heavy_model_inference=false`、`hot_path_spawn_per_query=false` 是硬门槛。

### 3.2.1 Mixed import correctness 完成

Mixed import 证据必须至少输出：

1. clean corpus 与 mixed nested corpus 的文档数、可搜索数、OCR backlog、
   failed、`non_resume`、`needs_review` 和 `resume_candidate` 计数。
2. indexed resume precision：
   `indexed_true_resumes / indexed_total`。
3. contamination count：进入 resume search index 的已知非简历数量。
4. resume completeness：
   `indexed_true_resumes / expected_true_resumes`。
5. mixed import overhead：mixed wall time、stage timings、content bytes、
   docs/s、MiB/s 相对 clean baseline 的变化。
6. H-tier/resource budget、隐私字段、benchmark layer
   `public_synthetic` / `private_calibration` / `blind_holdout`。

第一阶段优先 precision 和 contamination。Completeness/recall 的优化必须以
precision 不下降和 contamination 不越线为前提。

### 3.3 GUI/manual/Codex 闭环完成

GUI 和手工验证必须能完成：

1. 选择本地 root。
2. 查看 Level1/2/3 可见性状态。
3. 执行 keyword、field/filter、hybrid 查询。
4. 查看结果详情和 snippet。
5. 看见 OCR/semantic 后台队列和暂停原因。
6. 导出脱敏 diagnostics。
7. 运行 Codex 可复现验证脚本，并得到只含 aggregate/hash 的证据。

## 4. 硬约束

1. 查询热路径只做检索、过滤、融合、bulk hydrate 和 snippet，不做 OCR、全文解析、重模型推理或 full merge。
2. GUI 只能通过 daemon IPC/diagnostics contract 访问系统，不直接读 SQLite、Tantivy index、vector sidecar 或私有 query set。
3. 性能证据只提交 aggregate、hash、schema version、bucket count 和红线结果。
4. breaking change 必须通过明确 schema/version/epoch 迁移进入，不保留隐式双写兼容。
5. `goal_complete` 不能由 D10K 或 D100K 单独触发，必须同时满足 D1M scale、soak/fault、GUI/manual 和公开合同校验。

## 5. Active Contract Authority

本目录是当前活跃后续目标的执行合同。执行顺序必须先完成 P0 contract，再进入性能、GUI 或平台实现。

P0 contract 包含：

1. `12_Review问题映射与修复责任.md`
2. `13_Loop_Engineering状态机.md`
3. `14_W0_W1验收矩阵与证据协议.md`
4. `05_Query_Benchmark与真实Query种子.md` 中的查询语义冻结
5. `06_Daemon_IPC与Diagnostics契约.md` 中的 versioned IPC/diagnostics contract

若 `GOAL.md`、本文件、review ledger、Loop state、W0/W1 矩阵之间出现冲突，后续实现必须停止并回到 Superpowers planning 或 local gstack-lite scope/plan gate，不得自行选择更容易通过的目标。

机器可读入口：

1. `ACTIVE_GOAL.toml` 锁定当前目标、active slice、允许路径和隐私边界。
2. `perf/acceptance-matrix.toml` 锁定 W0/W1/soak/fault/GUI 的红线。
3. `perf/loop-state.schema.json` 锁定长程 Loop 状态报告格式。
4. `perf/experiment-report.schema.json` 锁定 redacted experiment report 格式。
5. `perf/current-loop-state.json` 记录当前 active slice 的公开状态快照。
6. `python3 scripts/ci/check-performance-contracts.py` 是公开机器验收入口，并已接入 `.github/workflows/pr.yml`。

P1 synthetic smoke baseline 是 P0 之后的公开 harness 入口。它只能使用
synthetic/public fixture，输出 redacted smoke report 和 artifact manifest，证明
后续 D10K private calibration、resident daemon benchmark、profile 和 GUI/manual
证据可以被机器校验；它不得声明 W1、D10K/D100K/D1M、profile optimization、
scale gate 或 `goal_complete`。
