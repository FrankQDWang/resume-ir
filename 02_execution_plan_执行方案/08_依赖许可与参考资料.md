## 2. 许可治理规则

1. 所有依赖进入仓库前必须过 license check。
2. 禁止引入许可证不清晰的模型和词典。
3. 模型许可单独审查，不能只看框架许可。
4. OCR 语言包、PDF 工具、转换工具都要进入 SBOM。
5. 商业版依赖必须通过产品和法务确认。
6. 运行时分发采用 bundled-first，并保留 external override。默认产品体验
   应尽量随安装包提供可审查 runtime；高级用户仍可指定本机命令路径。
7. MIT 不是产品打包硬约束。若选择 GPL-family renderer/runtime 进入默认包，
   release distribution 可以切换到 GPL-3.0-or-later 或其他兼容许可证，但
   必须先完成 GPL-family 分发审查、许可证文本/source-offer、SBOM 和
   installer composition 审批。
8. 外部命令和 bundled runtime 都不能把第三方许可证当成仓库代码许可证；
   manifest 必须记录 exact version、checksum、license、reviewed 状态、
   安装来源、`runtime_distribution_mode` 和是否包含 runtime binary。

## 3. 版本锁定

| 对象 | 锁定方式 |
|---|---|
| Rust crates | `Cargo.lock` |
| worker 外部二进制 | checksum + manifest |
| 模型文件 | checksum + model manifest |
| 词典 | version + checksum |
| 安装包 | release checksum |
| 索引 schema | manifest schema version |

## 5. 安全审查重点

| 类别 | 检查项 |
|---|---|
| 文档解析 | 恶意 PDF/Word、zip bomb、路径穿越 |
| OCR | 临时文件泄露、超时、资源耗尽 |
| 模型 | 模型文件篡改、许可、推理输入日志 |
| 索引 | 明文 PII、删除残留、快照校验 |
| 诊断 | 脱敏、最小化、用户授权 |
| 更新 | 签名、checksum、回滚 |

## 6. 官方资料摘要

- Tantivy 官方 README 标明它是 Rust 编写的全文搜索库，具备 BM25、增量索引、多线程索引、mmap directory、fast fields，并支持 Linux、macOS、Windows。
- SQLite FTS5 官方文档覆盖全文查询、prefix、NEAR、column filters、boolean operators、tokenizers 等能力。
- HNSWlib 官方 README 标明其为 header-only C++ HNSW 实现，支持插入、更新、删除标记，且轻量无额外依赖。
- FAISS 官方文档说明其用于 dense vector 的高效相似搜索，可处理超出 RAM 的向量集，并支持压缩、非穷举搜索和磁盘存储索引。
- ONNX Runtime 官方文档提供 8-bit 量化、Transformer 动态量化建议、CPU/CoreML/DirectML 等执行路径。
- OCRmyPDF 官方文档说明其用于给扫描 PDF 添加可搜索 OCR 文本层。
- Apache Tika 官方资料说明其可从上千种文件类型中检测和抽取文本/元数据，但当前 3.x 需要 Java 11，适合做隔离 fallback。
- GitHub 官方文档说明 GitHub Actions 可用于 CI/CD，CODEOWNERS 可定义代码责任人并要求 owner approval。

## 7. 依赖升级策略

1. 每月例行依赖升级窗口。
2. 安全漏洞可走 hotfix。
3. parser、OCR、索引依赖升级必须跑 golden tests。
4. 向量和模型依赖升级必须跑 recall benchmark。
5. SQLite/SQLCipher 升级必须跑 migration 和 crash recovery。
6. 安装包依赖升级必须跑 Windows/macOS 安装升级卸载。

## 8. 不建议依赖清单

| 依赖类型 | 不建议原因 |
|---|---|
| 本地 Elasticsearch | 重、服务化、桌面资源控制差 |
| 本地 Milvus/Weaviate | 数据库部署思维过重，不适合轻内核 |
| Python 常驻主进程 | 资源、打包、依赖和崩溃隔离成本高 |
| Java 常驻主内核 | JVM 常驻成本和桌面打包复杂度较高 |
| 大模型端到端解析 | 慢、贵、不可解释、难离线压缩 |
| 全量 PDF OCR | CPU 灾难，必须先检测文本层 |
