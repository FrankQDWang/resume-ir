# 代码仓库与目录设计

## 1. 仓库形态

采用单仓库多 crate/多包结构。原因：

1. 核心模块耦合度高，需要统一版本。
2. CI 可以做跨平台矩阵。
3. 领域模型、接口、测试基准可共享。
4. 便于 CODEOWNERS 精细化。
5. 模型、工具、文档、安装包配置统一管理。

## 3. crates 说明

| crate | 责任 | 禁止依赖 |
|---|---|---|
| `core-domain` | Document、ResumeVersion、Section、Entity、错误码、ID | 禁止依赖具体存储和索引 |
| `config` | 配置加载、profile、路径规范 | 禁止依赖业务执行模块 |
| `fs-crawler` | 文件枚举、监听、路径规范化 | 禁止解析文档内容 |
| `ingest-scheduler` | 任务队列、优先级、背压、重试 | 禁止直接做解析/OCR |
| `parser-common` | 解析接口、ParseResult、版面结构 | 禁止依赖具体 parser |
| `parser-docx` | docx 文本、表格、页眉页脚抽取 | 禁止索引写入 |
| `parser-pdf` | PDF 文本层、页面信息、OCR 判定 | 禁止 OCR 执行 |
| `parser-fallback` | 调用隔离转换 worker | 禁止进入查询路径 |
| `ocr-client` | 与 OCR worker 通信、缓存键 | 禁止直接调用 UI |
| `text-normalizer` | 文本清洗、offset 映射 | 禁止抽取业务字段 |
| `sectionizer` | 简历段落识别 | 禁止写索引 |
| `extractor-rules` | 正则、词典、字段推导 | 禁止模型 session 管理 |
| `extractor-model` | NER/分类模型推理封装 | 禁止 OCR |
| `embedder` | embedding 生成、batch、量化 | 禁止查询排序 |
| `meta-store` | 元数据、任务、状态、WAL、迁移 | 禁止全文搜索逻辑 |
| `index-fulltext` | 倒排索引 schema、writer、reader | 禁止业务字段推导 |
| `index-vector` | 向量索引、快照、增量更新 | 禁止文本解析 |
| `search-planner` | 查询解析、过滤计划、召回计划 | 禁止直接访问文件系统 |
| `rank-fusion` | RRF、加权融合、去重折叠 | 禁止索引写入 |
| `snippet` | 片段、高亮、offset 映射 | 禁止字段抽取 |
| `privacy` | 脱敏、hash、加密密钥接口 | 禁止保存明文日志 |
| `diagnostics` | 指标、doctor、诊断包 | 禁止输出敏感明文 |
| `ipc-api` | gRPC/HTTP API | 禁止包含业务实现 |
| `daemon` | 进程组装、生命周期 | 只依赖接口和实现模块 |
| `cli` | 命令行 | 禁止绕过 daemon 直接改索引 |

## 4. workers 说明

### 4.1 `ocr-worker`

职责：

1. PDF 页面渲染。
2. 图片预处理。
3. OCR 调用。
4. 返回页级 OCR 结果。
5. 受主进程调度和资源限制。

进程隔离原因：OCR 对 CPU、内存、临时文件和第三方库依赖较重，崩溃时不能影响查询。

### 4.2 `doc-convert-worker`

职责：处理旧 `.doc` 或复杂格式 fallback。

要求：

1. 独立临时目录。
2. 超时 kill。
3. 输出受控格式。
4. 不读取无关路径。
5. 日志脱敏。

### 4.3 `model-worker`

两种模式：

1. 小模型内嵌 daemon，使用 session pool。
2. 重模型独立 worker，批量处理。

默认先内嵌小模型，若出现内存或崩溃问题再拆 worker。

## 5. proto 设计

拆三个 proto：

| 文件 | 服务 |
|---|---|
| `resume_search.proto` | SearchService、DocumentService |
| `ingest.proto` | IngestService、TaskService |
| `status.proto` | HealthService、DiagnosticsService |

不要把所有接口塞进一个巨大服务。

每个词典必须有：

1. 版本号。
2. 来源说明。
3. 变更记录。
4. 测试样例。


## 8. 分支结构

```text
main              # 永远可发布
release/x.y       # 发布稳定分支
feature/...       # 短生命周期功能分支
fix/...           # bugfix
experiment/...    # 实验，不允许长期存在
```

## 9. 版本号

建议版本号分开：

| 版本 | 示例 |
|---|---|
| 应用版本 | `1.4.2` |
| 索引 schema | `fts_schema_v3` |
| 元数据 schema | `meta_schema_v5` |
| 模型包版本 | `model_pack_2026_05` |
| 词典版本 | `dict_2026_05_30` |

程序升级不应强制模型升级；模型升级也不应强制程序升级，除非 schema 不兼容。
