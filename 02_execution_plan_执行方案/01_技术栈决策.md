# 技术栈决策

## 1. 总体结论

推荐主路线：

```text
Rust 本地守护进程
  + Tantivy 全文/字段倒排
  + SQLite/SQLCipher 元数据与任务状态
  + HNSWlib 默认向量索引
  + FAISS CPU 作为大规模压缩向量索引备选
  + ONNX Runtime 统一模型推理
  + Tesseract/OCRmyPDF 作为 OCR 支路
  + gRPC/本地 IPC 暴露给 UI 或 agent
```

不推荐主路线：

1. Elasticsearch 本地常驻。
2. Milvus/Weaviate 本地数据库化部署。
3. Python 常驻主进程。
4. Java/JVM 常驻主内核。
5. 查询时调用大模型解析简历。
6. 对所有 PDF 无脑 OCR。

## 2. 技术栈矩阵

| 层 | 首选 | 备选 | 说明 |
|---|---|---|---|
| 主语言 | Rust | C++ | Rust 在性能、内存安全、跨平台工程之间平衡最好 |
| 异步运行时 | Tokio | 标准线程池 | IPC、任务调度、状态接口适合 async |
| CPU 并行 | Rayon | 自定义线程池 | 解析、抽取、批量向量计算适合 work stealing |
| 全文/字段索引 | Tantivy | SQLite FTS5 / Lucene | 本地库形态、mmap、BM25、fast fields、跨 Win/macOS |
| 元数据 | SQLite | RocksDB | SQLite 足够稳，任务、状态、事务、配置都适合 |
| 本地加密 | SQLCipher | OS 磁盘加密 | 简历敏感信息建议应用级加密 |
| 向量索引默认 | HNSWlib | Annoy | HNSWlib 轻量、增量、可删除标记 |
| 大规模向量 | FAISS CPU | 自研 PQ | 超大规模和压缩时用 FAISS |
| 模型推理 | ONNX Runtime | 平台原生推理 | 统一 CPU/CoreML/DirectML |
| OCR | Tesseract | OCRmyPDF / 平台 OCR | Tesseract 主 OCR；OCRmyPDF 适合扫描 PDF 文本层 |
| PDF 文本抽取 | PDFium/MuPDF 封装 | Tika fallback | 主链路原生；复杂文件隔离 fallback |
| DOCX 解析 | 原生 XML/zip parser | Tika fallback | docx 是 zip+xml，主链路不应依赖 Office |
| 旧 DOC | 隔离转换 worker | Tika/LibreOffice fallback | 不放主进程 |
| IPC | gRPC over UDS/Named Pipe/loopback | HTTP/JSON | agent 调用优先 gRPC，调试保留 HTTP |
| UI | Tauri 或原生 UI | 无 UI，只 CLI | 不建议 Electron 做主壳 |
| 打包 | MSI/WiX、macOS pkg/dmg | zip portable | 企业环境优先安装包，开发调试用 portable |

## 3. 为什么是 Rust 主内核

理由：

1. 原生性能，适合百万级本地索引和 mmap 读取。
2. 内存安全，适合长期常驻守护进程。
3. Windows/macOS 跨平台构建成熟。
4. 与 Tantivy 自然配合。
5. 便于把 OCR、模型、转换这类重/不稳定组件隔离到 worker。

C++ 可以更极致，但团队治理、内存安全、跨平台构建和崩溃风险更高。Go 做守护进程舒服，但核心检索库和向量库大概率仍要 FFI，收益不如 Rust 统一。

## 4. 全文索引选型

### 首选：Tantivy

使用位置：

1. 文件名索引。
2. 全文索引。
3. section 文本索引。
4. 字段文本索引。
5. fast fields 数值/枚举/时间过滤。
6. facet 和聚合。

决策原因：

1. Rust 原生库，不是服务化产品。
2. 支持 BM25、短语、范围、facet、fast fields。
3. 支持 mmap directory、增量索引、多线程索引。
4. 支持 Linux/macOS/Windows。
5. 启动和嵌入式集成更适合桌面。

### SQLite FTS5 的位置

SQLite FTS5 不做主全文引擎，但可以做：

1. MVP fallback。
2. 小库单文件模式。
3. 元数据局部搜索。
4. 诊断工具搜索。

原因：百万级混合检索、字段权重、片段、高频写入、索引合并等能力上，专用倒排引擎更合适。

### 为什么不选 Elasticsearch/Meilisearch 做最终主内核

这类服务化产品功能完整，但桌面端问题明显：

1. 多一个常驻服务。
2. 资源不可控性更高。
3. 安装、升级、崩溃恢复更重。
4. 与本地 agent 的轻内核目标冲突。
5. 很难做到“低配机器优先稳定”。

Meilisearch 可用于原型，但不建议作为最终内核。

## 5. 向量索引选型

### 默认：HNSWlib

适用：

1. 10 万到 100 万简历。
2. 需要增量添加、更新、删除标记。
3. CPU-only 桌面环境。
4. 低运维复杂度。

参数初始建议：

| 参数 | Economy | Balanced | Turbo |
|---|---:|---:|---:|
| `M` | 16 | 32 | 32 |
| `ef_construction` | 100 | 200 | 300 |
| `ef_search` | 32-64 | 64-128 | 128-256 |
| 向量精度 | int8/fp16 | int8/fp16 | fp16/fp32 |

### 扩展：FAISS CPU

适用：

1. 300 万级或更大。
2. 向量内存必须压缩。
3. 需要 IVF/PQ/OPQ 等压缩方案。
4. 能接受更复杂的构建和调参。

建议路径：先 HNSWlib 跑通，再用 FAISS 做压缩向量索引替代或补充，不要第一天就把工程复杂度拉满。

## 6. 模型推理

统一转 ONNX，并按平台选择执行提供方：

| 平台 | 默认 | 可选 |
|---|---|---|
| Windows | CPU int8 | DirectML |
| macOS Intel | CPU int8 | 无强依赖 |
| macOS Apple Silicon | CPU int8 | CoreML EP |

使用策略：

1. 字段抽取模型必须小。
2. embedding 模型优先 384 或 512 维。
3. 查询时默认不跑重 reranker。
4. reranker 只允许 top10/top20 小批量。
5. 模型包与程序包分开发布。

## 7. OCR 与格式解析

### OCR

1. Tesseract 作为主 OCR 引擎。
2. OCRmyPDF 用于扫描 PDF 加文本层的批处理。
3. OCR worker 独立进程，防止崩溃污染主进程。
4. OCR 默认限流，不进入查询热路径。

### PDF

1. 先检测文本层。
2. 有文本层直接抽取。
3. 没有文本层再进入 OCR。
4. 分页 OCR，允许部分成功。

### Word

1. `.docx` 走原生 zip+xml 解析。
2. `.doc` 走隔离 worker 转换或兼容 fallback。
3. 转换 worker 必须有超时和沙箱目录。

## 8. 本地服务形态

进程建议：

```text
resume-daemon        # 主守护进程：查询、调度、索引、状态
resume-cli           # CLI：导入、查询、诊断、重建
resume-ocr-worker    # OCR 子进程：可选启动、可限流
resume-model-worker  # 模型子进程：可选，或内嵌 ONNX Runtime session pool
resume-ui            # 可选桌面 UI
```

主守护进程不能直接被 OCR 和旧 DOC 转换拖死。

## 9. 关键取舍

| 问题 | 决策 |
|---|---|
| 极致性能还是开发快 | 极致性能优先 |
| 本地库还是本地服务 | 本地库优先 |
| OCR 精度还是资源 | 默认资源优先，用户可切高精度 |
| 向量召回还是字段准确 | 强过滤字段优先，向量做召回补充 |
| 单体还是多进程 | 主内核单体，重任务 worker 隔离 |
| 一开始支持所有格式 | 支持主格式，异常格式可 fallback |
